# Mem0: Building Production-ready AI Agents with Scalable Long-term Memory

## 一句话总结

提出 Mem0 和 Mem0g 两种记忆架构：Mem0 通过 LLM 动态提取、合并和检索自然语言形式的对话记忆；Mem0g 在此基础上引入图结构记忆（实体-关系三元组），在 LOCOMO 长对话记忆基准上全面超越现有记忆系统，同时实现比全上下文方法低 91% 的 p95 延迟和超过 90% 的 token 成本节省。

## 研究背景与动机

### LLM 的固定上下文困境

LLM 虽然能生成流畅的上下文连贯回复，但其固定上下文窗口在持续多轮对话中面临根本性挑战：

1. **跨会话遗忘**：用户在一次会话中提到的偏好（如素食、无乳制品），在后续会话中可能被完全遗忘。
2. **上下文窗口终究不够**：即使 GPT-4（128K）、Claude 3.7 Sonnet（200K）、Gemini（10M+），也无法覆盖数周或数月的真实对话历史。
3. **注意力退化**：注意力机制对远距离 token 的利用效果下降（Guo et al., 2024; Nelson et al., 2024），长上下文中关键信息可能被淹没在无关内容中。
4. **全上下文成本高昂**：每次查询都传入完整对话历史，导致延迟和 token 成本随对话长度线性增长。

> **核心洞察**：扩展上下文窗口只是延迟了问题，而非解决。真正的解决方案需要选择性存储、整合关联概念、按需检索——模拟人类认知过程。

### 现有记忆系统的不足

论文系统对比了六类基线：
- 已建立的 LOCOMO 基准方法（LoCoMo, ReadAgent, MemoryBank, MemGPT, A-Mem）
- 开源记忆方案（LangMem）
- RAG 变体（多种 chunk 大小和 k 值）
- 全上下文处理
- 专有模型（OpenAI ChatGPT memory）
- 商业记忆平台（Zep）

这些方案要么检索精度不足（RAG），要么延迟过高（LangMem p95 达 60 秒），要么 token 开销巨大（Zep 图记忆超过 600K tokens）。

## 核心方法

### 2.1 Mem0：文本级记忆架构

Mem0 采用**增量处理范式**，每个消息对（用户消息 + 助手回复）经过两阶段流水线：

#### 提取阶段（Extraction）

输入：新消息对 $(m_{t-1}, m_t)$ + 对话摘要 $S$ + 最近 $m=10$ 条历史消息

通过 LLM 提取函数 $\phi(P)$ 从当前交换中提取一组显著记忆 $\Omega = \{\omega_1, \omega_2, ..., \omega_n\}$。

- **对话摘要 $S$**：异步生成，提供全局主题理解
- **近期消息序列**：提供细粒度时间上下文
- **双上下文策略**：全局语义 + 局部细节，确保提取既全面又精确

#### 更新阶段（Update）

对每个候选事实 $\omega_i$，系统：
1. 检索 top $s=10$ 个语义相似的记忆（向量嵌入搜索）
2. 将候选事实 + 相似记忆送入 LLM，通过**工具调用（Tool Call）**机制决定操作：
   - **ADD**：创建新记忆（无语义等价记忆）
   - **UPDATE**：扩充现有记忆（信息互补）
   - **DELETE**：删除矛盾记忆（新信息推翻旧信息）
   - **NOOP**：无需修改

> 关键设计：不依赖单独分类器，而是利用 LLM 的推理能力直接选择操作。这使得系统能处理复杂的语义关系（如部分重叠、时间冲突、粒度差异）。

#### 实现配置

| 参数 | 值 |
|------|-----|
| 近期消息窗口 $m$ | 10 |
| 相似记忆检索数 $s$ | 10 |
| 推理引擎 | GPT-4o-mini |
| 向量数据库 | 密集嵌入 + 语义相似度搜索 |

### 2.2 Mem0g：图增强记忆架构

Mem0g 在 Mem0 基础上引入**有向标记图** $G = (V, E, L)$：

- **节点 $V$**：实体（如 Alice, San_Francisco）
- **边 $E$**：实体间关系（如 lives_in）
- **标签 $L$**：节点语义类型（如 Person, City）

每个实体节点包含：实体类型分类、语义嵌入向量 $e_v$、创建时间戳 $t_v$。

关系表示为三元组 $(v_s, r, v_d)$，其中 $v_s$ 为源节点，$v_d$ 为目标节点，$r$ 为关系标签。

#### 两阶段提取流水线

1. **实体提取器**：识别对话中的关键信息元素（人物、地点、对象、概念、事件、属性）
2. **关系生成器**：分析实体间的语义关联，生成关系三元组（如 `lives_in`、`prefers`、`owns`、`happened_on`）

关系生成不仅依赖显式陈述，还通过提示工程引导 LLM 推理隐含信息。

#### 存储与更新策略

- 新关系三元组计算源/目标实体嵌入
- 语义相似度阈值 $t$ 判断节点是否已存在
- **冲突检测机制**：识别新旧关系冲突，LLM 判断哪些关系应标记为过时（非物理删除，保留用于时间推理）

#### 双路径检索

| 方法 | 适用场景 | 机制 |
|------|----------|------|
| **实体中心检索** | 目标实体明确的问题 | 识别查询中的关键实体 → 定位图节点 → 探索入/出边关系 → 构建子图 |
| **语义三元组检索** | 概念性、开放式查询 | 将完整查询编码为嵌入向量 → 与所有关系三元组匹配 → 返回相似度超阈值的三元组 |

#### 技术栈

- 图数据库：Neo4j
- 提取与更新：GPT-4o-mini + Function Calling
- 嵌入：密集向量 + 语义相似度

## 实验结果

### 3.1 数据集：LOCOMO

- 10 段扩展对话，每段约 600 轮对话、26,000 tokens
- 每段对话伴随约 200 个问题（含标准答案）
- 问题类型：单跳（single-hop）、多跳（multi-hop）、时序（temporal）、开放域（open-domain）
- 排除对抗性问题（无标准答案）

### 3.2 评估指标

| 类别 | 指标 | 说明 |
|------|------|------|
| **性能** | F1 Score | 词汇重叠 |
| | BLEU-1 (B1) | 词汇相似度 |
| | LLM-as-a-Judge (J) | GPT-4o 评判事实准确性、相关性、完整性 |
| **部署** | Token Consumption | 检索上下文 token 数 |
| | Search Latency | 记忆/块检索时间 |
| | Total Latency | 检索 + 生成总时间 |

> J 指标进行 10 次独立运行，报告均值 ± 标准差。

### 3.3 性能对比（按问题类型）

#### 单跳问题（Single-Hop）

| 方法 | F1 | B1 | J |
|------|-----|-----|-----|
| A-Mem* | 20.76 | 14.90 | 39.79 |
| LangMem | 35.51 | 26.86 | 62.23 |
| Zep | 35.74 | 23.30 | 61.70 |
| OpenAI | 34.30 | 23.72 | 63.79 |
| **Mem0** | **38.72** | **27.13** | **67.13** |
| Mem0g | 38.09 | 26.03 | 65.71 |

Mem0 在单跳检索上全面领先。Mem0g 略有下降，说明图结构对简单单事实检索的增益有限。

#### 多跳问题（Multi-Hop）

| 方法 | F1 | B1 | J |
|------|-----|-----|-----|
| A-Mem* | 9.22 | 8.81 | 18.85 |
| LangMem | 26.04 | 22.32 | 47.92 |
| Zep | 19.37 | 14.82 | 41.35 |
| OpenAI | 20.09 | 15.42 | 42.92 |
| **Mem0** | **28.64** | **21.58** | **51.15** |
| Mem0g | 24.32 | 18.82 | 47.19 |

Mem0 在多跳推理上优势明显。意外的是 Mem0g 未带来提升，论文推测图结构在多步整合任务中可能引入冗余开销。

#### 开放域问题（Open-Domain）

| 方法 | F1 | B1 | J |
|------|-----|-----|-----|
| Zep | **49.56** | **38.92** | **76.60** |
| Mem0 | 47.65 | 38.72 | 72.93 |
| **Mem0g** | 49.27 | 40.30 | **75.71** |

Zep 在开放域上略领先（J 76.60 vs Mem0g 75.71，差距 0.89），但 Mem0g 紧随其后。说明 Zep 在外部知识整合上有微弱优势。

#### 时序推理（Temporal）

| 方法 | F1 | B1 | J |
|------|-----|-----|-----|
| A-Mem* | 35.40 | 31.08 | 49.91 |
| Zep | 42.00 | 34.53 | 49.31 |
| OpenAI | 14.04 | 11.25 | 21.71 |
| Mem0 | 48.93 | **40.51** | 55.51 |
| **Mem0g** | **51.55** | 40.28 | **58.13** |

**Mem0g 在时序推理上表现最强**，图结构显式建模时间关系（事件序列、相对顺序）带来显著增益。OpenAI 表现最差，因其记忆提取遗漏时间戳。

### 3.4 总体性能与延迟对比

| 方法 | 记忆/块 Token 数 | Search p50 | Search p95 | Total p50 | Total p95 | Overall J |
|------|-----------------|------------|------------|-----------|-----------|-----------|
| RAG (best) | 16384 | 0.247s | 0.746s | 0.829s | 1.729s | 60.97 |
| Full-context | 26031 | - | - | 9.870s | **17.117s** | **72.90** |
| A-Mem | 2520 | 0.668s | 1.485s | 1.410s | 4.374s | 48.38 |
| LangMem | 127 | 17.99s | 59.82s | 18.53s | 60.40s | 58.10 |
| Zep | 3911 | 0.513s | 0.778s | 1.292s | 2.926s | 65.99 |
| OpenAI | 4437 | - | - | 0.466s | 0.889s | 52.90 |
| **Mem0** | **1764** | **0.148s** | **0.200s** | **0.708s** | **1.440s** | **66.88** |
| **Mem0g** | **3616** | 0.476s | 0.657s | 1.091s | 2.590s | **68.44** |

**关键发现**：
- Mem0 的 **p95 延迟仅 1.44 秒**，比全上下文方法（17.12 秒）低 **91.6%**
- Mem0 的 **Search p95 仅 0.20 秒**，是所有方法中最快的
- Mem0 平均仅使用 **1,764 tokens** 的记忆上下文，比全上下文（26,031）节省 **93.2%**
- Mem0g 在总体 J 上最高（68.44），仅落后于计算上不可行的全上下文方法

### 3.5 Token 开销与构建时间

| 方法 | 每对话平均 Token | 备注 |
|------|-----------------|------|
| Mem0 | ~7,000 | 自然语言记忆表示 |
| Mem0g | ~14,000 | 图记忆（节点 + 关系） |
| Zep | >600,000 | 每个节点缓存完整摘要 + 边存储事实，高度冗余 |
| 原始对话 | ~26,000 | 无记忆抽象 |

**Zep 的致命问题**：
- Token 开销是原始对话的 20 倍以上
- 记忆添加后**立即检索常常失败**，需等待数小时后才能获得准确结果
- 图构建涉及多轮异步 LLM 调用和大量后台处理，不适用于实时应用
- Mem0 图构建即使在最坏情况下也在 **1 分钟内完成**

## 工程实践

### 生产级部署考量

1. **增量处理**：每轮对话实时提取和更新记忆，无需批处理或离线重建
2. **异步摘要**：对话摘要独立刷新，不阻塞主处理流水线
3. **向量检索优化**：密集嵌入 + 近似最近邻搜索，确保亚秒级检索
4. **LLM 调用最小化**：提取和更新各需一次 LLM 调用，检索无需 LLM
5. **可配置参数**：近期消息窗口、相似记忆数、相似度阈值均可调

### 与全上下文方法的权衡

| 维度 | 全上下文 | Mem0 | Mem0g |
|------|----------|------|-------|
| 准确率（J）| 72.90（最高）| 66.88 | 68.44 |
| p95 延迟 | 17.1s（不可接受）| 1.44s | 2.59s |
| Token 成本 | 26K/查询 | 1.8K/查询 | 3.6K/查询 |
| 可扩展性 | 随对话长度恶化 | 恒定 | 恒定 |
| 适用场景 | 短对话、离线分析 | 实时交互 Agent | 复杂关系推理 |

## 与 δ-mem 的对比

Mem0 和 δ-mem 代表了 LLM 记忆机制的**两条根本不同的路径**：

| 维度 | Mem0（文本记忆 + 图结构） | δ-mem（参数级在线状态） |
|------|--------------------------|------------------------|
| **记忆存储位置** | 外部数据库（向量库 + 图数据库） | 模型内部（8×8 状态矩阵） |
| **记忆形式** | 自然语言文本 + 实体-关系三元组 | 压缩的关联记忆状态（浮点矩阵） |
| **提取方式** | LLM 显式提取对话中的显著信息 | 隐式：每个 token 的 key-value 自动写入状态 |
| **检索方式** | 向量语义搜索 + 图遍历 | 状态矩阵读出，与历史长度无关 |
| **更新机制** | LLM 判断 ADD/UPDATE/DELETE/NOOP | 门控 delta-rule 在线更新 |
| **与骨干关系** | 完全解耦，可搭配任意 LLM | 深度耦合，低秩修正直接注入注意力 |
| **可训练参数** | 无（纯提示工程 + 检索） | 4.87M（骨干的 0.12%） |
| **延迟特性** | 检索延迟 + LLM 生成延迟 | 几乎无额外延迟（状态读出为矩阵乘法） |
| **可解释性** | 高：记忆为人类可读文本 | 低：状态矩阵为黑盒压缩表示 |
| **跨模型迁移** | 容易：记忆库与模型无关 | 困难：状态绑定特定骨干架构 |
| **记忆容量** | 理论上无限（数据库可扩展） | 受限于 8×8 矩阵的压缩容量 |
| **最佳场景** | 生产级 Agent、多用户系统、需要可解释记忆 | 单模型部署、极致延迟敏感、资源受限环境 |

### 互补性

两条路径并非互斥，而是互补：
- **Mem0 解决"存什么"和"怎么找"**：通过 LLM 提取高质量记忆，通过向量/图检索精确召回
- **δ-mem 解决"怎么记"和"怎么用"**：通过紧凑状态矩阵实现零延迟记忆读出，直接修正注意力

未来可能的方向：Mem0 负责长期、结构化、可解释的外部记忆；δ-mem 负责短期、在线、低延迟的内部记忆引导。两者结合可能同时获得 Mem0 的可扩展性和 δ-mem 的实时性。

## 优点与局限

### 优点

1. **全面领先现有系统**：在 LOCOMO 上全面超越所有基线（单跳、多跳、时序）
2. **极致延迟优化**：p95 延迟 1.44s，比全上下文低 91%，比 LangMem 低 97%
3. **Token 成本节省**：比全上下文节省 90%+，比 Zep 节省 98%+
4. **生产就绪**：增量处理、异步摘要、实时可用，无需等待后台构建
5. **可解释性强**：记忆为人类可读的自然语言文本，便于调试和审计
6. **模型无关**：可搭配任意 LLM，不限定特定骨干

### 局限与开放问题

1. **开放域略逊于 Zep**：J 差距 0.89，说明外部知识整合仍有提升空间
2. **Mem0g 多跳未达预期**：图结构在多跳整合中未带来增益，可能因图遍历开销
3. **依赖 LLM 进行提取和更新**：每次对话至少 2 次 LLM 调用，成本不可忽视
4. **图构建复杂度**：Mem0g 的实体提取和关系生成增加了系统复杂度
5. **未探索与长上下文的联合使用**：论文未测试 Mem0 + 长上下文窗口的组合效果
6. **仅在 LOCOMO 上评估**：缺乏跨领域（医疗、教育、企业支持）的验证
