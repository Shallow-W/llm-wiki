# 文本记忆（Text Memory）

## 定义

文本记忆是将对话历史中的显著信息提取为**自然语言陈述句**（事实、偏好、事件等），存储在外部数据库中，并在需要时通过语义相似度检索召回，作为 LLM 的上下文补充。

## 核心思想

不将完整对话历史传入上下文窗口，而是让 LLM 充当"记忆编辑"，从每轮对话中提炼出关键事实，以简洁的自然语言形式持久化存储：

> **把长对话压缩为精炼的事实集合，每次查询只召回最相关的几条事实，而非整段对话。**

## 关键属性

### 记忆形式

- 自然语言句子或短语
- 人类可读、可解释、可审计
- 通常带时间戳和来源标注

### 典型结构

```
[时间戳] 用户A 喜欢印度菜，尤其喜欢黄油鸡
[时间戳] 用户B 住在旧金山，在 Google 工作
[时间戳] 用户A 对乳糖不耐受
```

### 操作流程

| 阶段 | 动作 | 执行者 |
|------|------|--------|
| **提取** | 从消息对中识别显著信息 | LLM |
| **更新** | 与现有记忆比对，决定 ADD/UPDATE/DELETE/NOOP | LLM + 向量检索 |
| **检索** | 根据查询语义召回 top-k 相关记忆 | 向量相似度搜索 |
| **生成** | 将检索到的记忆作为上下文，生成回复 | LLM |

## 与 RAG 的区别

| 维度 | 文本记忆 | RAG |
|------|----------|-----|
| **存储单位** | 精炼的事实句子 | 原始文本块（chunk） |
| **信息密度** | 高：每 token 承载更多语义 | 低：包含大量冗余和无关信息 |
| **更新粒度** | 细：可单独更新单个事实 | 粗：需替换整个 chunk |
| **一致性维护** | 可检测和解决矛盾 | 难以处理 chunk 内矛盾 |
| **可解释性** | 高：每条记忆独立可读 | 中：需阅读整个 chunk |
| **检索精度** | 高：语义匹配事实 | 中：可能召回不相关段落 |

## 代表系统

| 系统 | 特点 |
|------|------|
| **Mem0** | LLM 提取 + 向量检索 + 工具调用更新，LOCOMO 上全面领先 |
| **MemoryBank** | 文本记忆 + 类人类遗忘机制（强化-衰减） |
| **LoCoMo** | 短期摘要 + 长期观察（observations）+ 时序事件图 |
| **A-Mem** | Agentic 笔记互联，动态演化记忆结构 |

## 优势与局限

### 优势

- **高信息密度**：比原始文本节省 90%+ token
- **人类可读**：便于调试、审计和人工修正
- **精确更新**：可针对单个事实进行增删改
- **低延迟**：检索仅需向量相似度搜索，亚秒级
- **模型无关**：不依赖特定 LLM 架构

### 局限

- **提取质量依赖 LLM**：提取不准确会导致记忆错误
- **关系隐含化**：实体间关系隐藏在文本中，不利于多跳推理
- **结构化查询困难**：难以回答"A 和 B 的共同朋友是谁"类问题
- **每次对话需 LLM 调用**：提取和更新各需一次 LLM 调用

## 与图记忆的互补

Mem0 论文的实验结果表明：

- **文本记忆**在单跳（+1.4% J）和多跳（+3.9% J）上优于图记忆
- **图记忆**在时序推理（+2.6% J）上优于文本记忆

这提示**混合架构**可能是最佳方案：用文本记忆处理日常事实和简单查询，用图记忆增强时序和关系推理。

## 与 δ-mem 的对比

| 维度 | 文本记忆（Mem0） | δ-mem |
|------|-----------------|-------|
| 存储位置 | 外部数据库 | 模型内部状态矩阵 |
| 形式 | 自然语言句子 | 8×8 浮点矩阵 |
| 可解释性 | 高 | 低 |
| 延迟 | 检索延迟 + 生成 | 几乎无额外延迟 |
| 容量 | 理论上无限 | 受矩阵大小限制 |
| 训练需求 | 无 | 需微调（4.87M 参数） |
