# Context Rot（上下文腐烂）

## 定义

**Context Rot**（上下文腐烂/退化）是指大型语言模型（LLM）在处理长输入时，即使**任务复杂度保持不变**，仅增加输入 token 长度就会导致性能非均匀退化的现象。该术语由 Chroma 团队（Hong et al., 2025）在 2025 年 7 月的技术报告中首次系统提出和验证。

核心洞察：**LLM 不会均匀地使用其上下文**——第 10,000 个 token 的处理可靠性显著低于第 100 个 token，即使两者承载的信息重要性相同。

## 关键属性

### 1. 非均匀性（Non-uniformity）

性能退化不是线性的，而是在某些"临界点"后急剧下降。不同模型、不同任务类型的退化曲线形态各异。

### 2. 任务无关性（Task Independence）

Context Rot 在最简单的任务中也能观察到——包括：
- 非词汇匹配的句子检索（Needle in a Haystack 扩展）
- 精确文本复现（Repeated Words）
- 对话历史中的事实检索（LongMemEval）

这意味着退化不是由任务内在难度导致的，而是由**输入长度本身**导致的。

### 3. 多因素交互（Multi-factor Interaction）

Context Rot 的严重程度受多个因素交互影响：

| 因素 | 影响方向 |
|------|----------|
| **Needle-Question 相似度** | 相似度越低，退化越快 |
| **干扰项（Distractors）** | 干扰项越多、越"逼真"，退化越严重 |
| **Needle-Haystack 相似度** | 结果不一致，但存在非均匀影响 |
| **Haystack 结构** | 逻辑连贯的 haystack 反而损害性能（反直觉） |
| **信息位置** | 在 Repeated Words 中，信息越靠后越容易丢失 |

### 4. 模型家族差异（Model Family Differences）

不同模型家族应对 Context Rot 的策略不同：

| 家族 | 典型行为 | 代表表现 |
|------|----------|----------|
| **Claude** | 保守弃权 | 不确定时明确拒绝回答，幻觉率低 |
| **GPT** | 自信幻觉 | 即使错误也生成自信回答，幻觉率高 |
| **Gemini** | 随机退化 | 长序列中出现无意义随机输出 |
| **Qwen** | 任务放弃 | 达到一定长度后直接不尝试任务 |

## 与相关概念的区别

### Context Rot vs 上下文窗口限制

- **上下文窗口限制**：模型根本"看不到"超出窗口的内容（硬限制）
- **Context Rot**：模型"能看到"所有内容，但**无法可靠处理**（软退化）

即使上下文窗口扩展到百万 token，Context Rot 依然存在。

### Context Rot vs 注意力稀释（Attention Dilution）

- **注意力稀释**：通常指注意力权重在长序列中被分散，每个位置获得的注意力减少
- **Context Rot**：范围更广，包括注意力机制之外的因素（如结构影响、干扰项放大、输出长度限制等）

注意力稀释可能是 Context Rot 的机制之一，但不是唯一机制。

### Context Rot vs 多轮迷失（Multi-turn Lost）

- **多轮迷失**（Laban et al., 2025）：特指多轮对话中模型丢失早期信息的现象
- **Context Rot**：更一般化，适用于任何长输入场景，不限于对话

两者互补：多轮对话中的迷失可以看作是 Context Rot 在对话场景下的具体表现。

## 实验证据

### 来源

主要实验证据来自 [[source-context-rot|Context Rot 技术报告]]，涵盖 18 个主流 LLM 的六个控制实验：

1. **Needle-Question 相似度**：低相似度配对的性能退化更快
2. **干扰项影响**：单个干扰项即可降低性能，4 个干扰项显著加剧退化
3. **Needle-Haystack 相似度**：语义相似度对性能有非均匀影响
4. **Haystack 结构**：打乱的 haystack 反而比连贯的 haystack 表现更好
5. **LongMemEval**：完整对话历史（113k tokens）vs 聚焦输入（300 tokens），所有模型在完整输入上性能下降
6. **Repeated Words**：最简单的文本复现任务，长序列下也出现位置错误和随机输出

## 对 LLM 架构的启示

### 为什么简单扩展上下文不够

Context Rot 揭示了 Transformer 注意力机制的根本局限：

1. **二次计算复杂度**：标准注意力的计算量随序列长度二次增长，限制了可处理的上下文规模
2. **注意力模式固化**：模型学习到的注意力模式可能不适应超长序列
3. **位置编码衰减**：绝对或相对位置编码在极长距离上可能失效
4. **输出长度耦合**：自回归生成中，输出长度也是输入的一部分，长输出进一步加剧问题

### 应对策略谱系

| 策略类型 | 代表方法 | 原理 | 局限 |
|----------|----------|------|------|
| **外部检索（RAG）** | BM25, Dense Retrieval | 只将相关片段放入上下文 | 检索噪声、相关片段可能遗漏 |
| **上下文压缩** | LLMLingua-2, Selective Context | 压缩或筛选上下文 | 信息损失、压缩本身引入错误 |
| **在线记忆状态** | [[δ-mem-在线关联记忆|δ-mem]], Titans | 将历史压缩到固定大小的动态状态 | 状态容量有限 |
| **参数化记忆** | LoRA, Context2LoRA | 将记忆编码到参数中 | 静态的，无法动态演化 |
| **稀疏注意力** | Longformer, BigBird | 降低注意力计算复杂度 | 可能丢失长距离依赖 |
| **线性注意力** | Mamba, RWKV, Kimi Linear | 用线性复杂度替代二次注意力 | 表达能力与标准注意力的差距 |

## 对应用开发的启示

### 上下文工程（Context Engineering）

Context Rot 表明，**信息如何呈现**比**信息是否存在**更重要：

1. **减少干扰项**：在 RAG 中，语义相似的文档片段应被更严格地去重或重排序
2. **结构化呈现**：虽然连贯结构在实验中表现更差，但这可能意味着需要新的结构化方法（如显式标记重要信息）
3. **位置优化**：关键信息应尽可能放在上下文的前部
4. **分块处理**：将长任务分解为多个短上下文调用

### 可靠性评估

在生产环境中部署长上下文应用时：
- 不应仅以 NIAH 分数判断模型的长上下文能力
- 需要在实际数据分布上测试语义匹配、干扰项容忍度等
- 考虑模型家族的特定行为（Claude 的保守性 vs GPT 的自信幻觉）

## 开放问题

1. **机制不明**：Context Rot 的根本原因尚未完全理解。是注意力机制的固有限制？还是训练数据分布的偏差？还是位置编码的衰减？
2. **与模型规模的关联**：更大的模型是否能更好地抵抗 Context Rot？
3. **与架构的关联**：非 Transformer 架构（如 Mamba、RWKV）是否同样受 Context Rot 影响？
4. **动态阈值**：是否存在一个"安全上下文长度"，在此之内 Context Rot 可以忽略？这个阈值是否因任务而异？
5. **与输出长度的交互**：Repeated Words 实验显示输入-输出长度耦合会加剧问题，但具体机制尚不清楚
