# Adapter-Revision Path（适配器修订路径）

## 定义

Adapter-Revision Path 是 [[mint-managed-infrastructure-training-serving-millions-llms|MinT]] 论文提出的核心设计模式：在 LoRA 后训练和服务系统中，基座模型保持常驻（resident），只有导出的 LoRA 适配器修订版（adapter revision）作为行为承载载荷在训练、rollout、评估、服务和回滚之间流转。它将"哪个模型变体在运行"这个问题从"移动完整检查点"转变为"选择和加载一个紧凑的适配器"。

## 关键属性

### 适配器修订版（Adapter Revision）

- **定义**：LoRA 适配器在特定训练步的冻结、导出快照，以服务张量布局（PEFT 格式）存储
- **不是**训练迭代或传输事件，而是可执行的 LoRA 负载
- 包含：适配器张量、秩、目标模块、基座模型兼容性元数据
- 不包含：合并的基座权重、优化器状态、rank 局部训练文件

### 策略记录（Policy Record）

- **定义**：服务拥有的生命周期状态，使适配器修订版可复现、可重载、可回滚
- 包含：基座版本名称、LoRA 配置（秩、目标模块）、最新训练检查点位置、rollout 记录、导出的适配器修订版列表

### 三种路径对比

| 路径 | 训练→服务边界移动的对象 | 特点 |
|------|--------------------------|------|
| 全量微调 | 完整检查点 $W_i$（每个变体一个完整模型） | 开销最大 |
| LoRA 合并 | 合并检查点 $W'_i = W + L_i$ | 减少训练内存但仍然移动完整模型 |
| **MinT 适配器修订路径** | 仅适配器 $L_i$（<1% 基座大小） | 开销最小，基座常驻 |

### 生命周期阶段

1. **Rollout**：在常驻基座 + 选定适配器修订版上采样轨迹
2. **Update**：为轨迹重新计算概率，应用目标函数，更新适配器
3. **Export**：冻结训练检查点为固定适配器修订版（PEFT 格式，服务布局）
4. **Evaluate**：使用固定修订版评估
5. **Serve**：通过缓存层（目录→CPU→GPU）加载并服务
6. **Rollback**：选择历史修订版回滚

### 服务端四问

每次更新后，系统需要回答：

1. 哪个基座部署能运行此适配器？
2. 哪个训练检查点可恢复训练？
3. 哪个固定适配器修订版用于 rollout/评估/服务？
4. 适配器字节现在在哪（GPU batch / CPU 缓存 / 共享存储）？

## 核心优势

- **交接开销极低**：Qwen3-4B 上加速 18.3 倍，Qwen3-30B 上加速 2.85 倍
- **适配器体积小**：rank-1 设置下仅约基座的 0.1%
- **支持并发**：多个策略可共享同一基座分配，峰值内存不变
- **可复现性**：每个行为都绑定到特定的适配器修订版
