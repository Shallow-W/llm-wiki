# 工具集成推理（Tool-Integrated Reasoning, TIR）

## 定义

让 LLM 在推理过程中动态调用外部工具（如代码解释器、搜索引擎、API），并将工具返回结果作为后续推理的上下文，形成"自然语言推理 ↔ 工具执行"交替进行的混合推理范式。

## 核心思想

传统 LLM 推理是纯文本的"思维链"（Chain-of-Thought）：

```
问题 → [内部推理] → 答案
```

工具集成推理扩展为"人机混合"模式：

```
问题 → [推理 t_1] → 工具调用 c_1 → [执行 f_1] → [推理 t_2] → 工具调用 c_2 → [执行 f_2] → ... → 答案
```

**关键洞察**：LLM 擅长高层推理规划，但容易在精确计算、符号操作、大规模数据处理上出错。外部工具恰好弥补这些短板。两者结合产生 1+1>2 的效果。

## 技术实现

### 多轮交互协议

Agent0 中采用的"停-走"（stop-and-go）策略：

1. 模型生成推理文本 $t_1$ 和代码 $c_1$（包裹在 ` ```python...``` ` 中）
2. 检测到代码块后暂停生成
3. 代码在隔离沙箱中执行，捕获 stdout/stderr 作为 $f_1$
4. 执行结果（前缀 ` ```output...``` `）追加到对话历史
5. 模型基于更新后的历史继续生成
6. 重复直到输出最终答案（`\boxed{...}`）

### 轨迹格式

完整的 TIR 轨迹是一个交错的序列：

$$\tau = [t_1, c_1, f_1, t_2, c_2, f_2, \dots, t_n, o]$$

其中 $t_i$ 是文本推理，$c_i$ 是代码调用，$f_i$ 是执行反馈，$o$ 是最终答案。

## TIR 在 Agent0 中的双重角色

### 角色一：执行 Agent 的问题解决工具

执行 Agent 使用代码解释器来：
- 验证数学计算（避免算术错误）
- 执行符号操作（代数化简、方程求解）
- 数值模拟（蒙特卡洛估计、迭代算法）
- 数据结构操作（排列组合、图算法）

### 角色二：课程 Agent 的能力杠杆

更重要的是，工具集成**改变了课程 Agent 可生成任务的空间**：

| 无工具 | 有工具 |
|--------|--------|
| 纯符号/逻辑问题 | 需要数值计算的问题 |
| 简单算术可解决的问题 | 大规模计算问题 |
| 闭式解问题 | 需要迭代/模拟的问题 |
| 静态问题 | 动态/交互式问题 |

课程 Agent 通过 $R_{\text{tool}}$ 奖励显式激励生成需要工具使用的任务，从而不断推动任务复杂度边界。

## TIR 的研究维度

当前 TIR 研究主要沿三个维度展开：

### 1. 稳定性（Stability）

多轮 RL 训练 notoriously 不稳定。解决方案：
- **ASPO** (Lin & Xu, 2025)：理论保证和梯度过滤
- **SimpleTIR** (Xue et al., 2025)：端到端 RL，处理空轮次
- **Agent0 ADPO**：模糊感知动态策略优化

### 2. 泛化性（Generalization）

工具使用技能能否跨域迁移：
- Chen et al. (2025) 证明工具使用技能具有跨域可迁移性
- Agent0 证明数学推理中培养的工具使用能力可迁移到通用推理

### 3. 复杂度（Complexity）

处理长程规划和复杂交互：
- **长程规划**：Plan-and-Act (Erdogan et al., 2025), Beyond Ten Turns (Gao et al., 2025)
- **记忆管理**：Memory-R1 (Yan et al., 2025)
- **交互效率**：OTC (Wang et al., 2025a)

## TIR 与纯文本推理的对比

| 维度 | 纯文本推理 | 工具集成推理 |
|------|-----------|-------------|
| 计算准确性 | 易出错 | 精确（代码执行） |
| 可处理问题范围 | 受限于模型知识 | 扩展至需要外部计算的问题 |
| 推理链长度 | 单轮或固定轮次 | 动态、自适应 |
| 训练复杂度 | 相对简单 | 需要处理工具调用和执行反馈 |
| 计算成本 | 仅推理 | 推理 + 代码执行 |
| 可解释性 | 完整思维链可见 | 思维链 + 执行记录 |

## 在 Agent0 中的实验效果

工具集成对 Agent0 的关键贡献：

- **vs 无工具方法**：Agent0（有工具）在 Qwen3-4B 上超越 R-Zero（无工具）3.4%（MATH）
- **vs 仅验证工具**：Agent0 超越 Absolute Zero（工具仅用于验证）6.1%（MATH）
- **工具使用进化**：3 次迭代中平均工具调用从 1.65 次增至 2.60 次

结论：**优势不仅在于"有工具"，更在于"学会如何使用工具"**——课程 Agent 学会生成需要工具的任务，执行 Agent 学会在推理中有效调用工具。

## 局限

1. **工具范围有限**：当前主要使用 Python 解释器，未充分利用搜索引擎、数据库、API 等
2. **执行安全风险**：代码执行需要沙箱隔离，增加系统复杂度
3. **延迟开销**：每次工具调用增加往返延迟，影响交互体验
4. **错误传播**：工具执行错误可能误导后续推理
5. **训练不稳定**：多轮交互的 RL 训练比单轮更难以稳定
