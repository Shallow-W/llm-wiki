# 微服务部署与路由优化概览

## 当前论点

微服务/AI 服务在数据中心和边缘网络中的高效部署，核心挑战在于**部署（Deployment）**、**路由（Routing）**、**资源分配（Resource Allocation）** 三个子问题的深度耦合：部署决定请求的路由路径，路由反过来影响部署效果，而资源分配贯穿始终。该领域的研究趋势是从**独立优化**走向**联合优化**，从**静态配置**走向**动态适应**，从**通用指标**走向**部署导向评估**。

---

## 1. 问题全景：部署-路由-资源分配的三元耦合

### 1.1 三个子问题是什么

| 子问题 | 决策内容 | 核心矛盾 |
|--------|----------|----------|
| **部署（Deployment）** | 把哪些服务实例放在哪些节点上 | 集中部署（减少通信延迟）vs. 分散部署（减少排队延迟） |
| **路由（Routing）** | 请求到达后转发到哪个实例 | 最短路径 vs. 负载均衡 vs. 端到端延迟最优 |
| **资源分配（Resource Allocation）** | 每个节点分配多少 CPU/GPU/带宽 | 服务质量 vs. 能耗 vs. 资源公平性 |

### 1.2 为什么它们耦合

```
用户请求 → [路由决策] → 服务实例A → [路由决策] → 服务实例B → ... → 响应
                ↑                              ↑
           部署决定实例A在哪              部署决定实例B在哪
                ↓                              ↓
           资源分配决定A的处理能力      资源分配决定B的处理能力
```

- **部署 → 路由**：实例部署在哪里，决定了请求可以走哪些路径
- **路由 → 部署**：请求如何路由，决定了部署是否真的有效（部署了但没流量等于浪费）
- **部署 → 资源**：部署多少实例，直接消耗节点资源
- **路由 → 资源**：路由策略决定每个实例的负载，影响资源利用

> 💡 **关键洞察**：把耦合问题拆成独立子问题分别求解（如 Kubernetes 把部署和路由做成分离模块），只能做到局部最优。联合优化是近年来的主要研究方向。

### 1.3 论文中的体现

| 论文 | 联合优化的维度 | 方法 |
|------|---------------|------|
| [[gstc-zero-cost-proxy-nas\|GSTC]] | 架构设计 + 部署 + 路由 + 动态替换 | GSTC 代理 + JCQDA + LAMRA |
| [[joint-deployment-request-routing-microservice-tpds\|TPDS 2023]] | 部署 + 路由 | GMDA + RMPR 两阶段启发式 |
| [[joint-task-offloading-resource-allocation-model-placement-6g\|TSC 2024]] | 任务卸载 + 资源分配 + 模型放置 | Lyapunov + DA + MAB |

## 2. 延迟模型：一切优化的基础

### 2.1 端到端延迟的三个组成

任何请求在系统中经历的延迟都可以分解为：

$$T_{\text{端到端}} = T_{\text{通信}} + T_{\text{排队}} + T_{\text{处理}}$$

| 延迟类型 | 产生原因 | 受什么影响 |
|----------|----------|-----------|
| **通信延迟** | 数据在网络节点间传输 | 节点间的物理距离、网络带宽、数据大小 |
| **排队延迟** | 请求在队列中等待空闲服务台 | 到达率 vs. 服务率、实例数量 |
| **处理延迟** | 服务台实际处理请求 | 任务计算量、模型复杂度、CPU/GPU 性能 |

### 2.2 排队论：用数学刻画延迟

排队论是部署与路由优化的基础工具。核心思想：将服务实例建模为"服务台"，请求建模为"顾客"，用到达率和服务率计算延迟。

**你需要知道的三个排队模型**（从简到繁）：

#### M/M/1 — 最基础
- 1 个服务台，泊松到达，指数服务时间
- 平均排队长度：$L_q = \frac{\rho^2}{1-\rho}$，其中 $\rho = \lambda/\mu$ 为利用率
- 适用：单实例、简单场景

#### [[M/M/C 排队模型]] — 论文中最常用
- C 个并行服务台（即 C 个实例）
- 相同服务率，共享一个队列
- 关键：C 越大 → 排队越短（但部署成本越高）
- 适用：多实例部署（[[gstc-zero-cost-proxy-nas\|GSTC]] 的 JCQDA 用这个）

#### [[Jackson 排队网络]] — 处理服务链
- 多个 M/M/C 节点组成的网络
- 请求在节点间流动（按路由概率）
- 乘积形式解：总延迟 = 各节点延迟之和
- 适用：微服务调用图（[[joint-deployment-request-routing-microservice-tpds\|TPDS 2023]] 核心工具）

### 2.3 排队论给部署优化的直觉

> **部署更多实例（增大 C）** → 排队延迟↓ 处理能力↑，但部署成本↑
> **把依赖服务放在一起** → 通信延迟↓，但可能排队延迟↑（热点集中）
> **分散部署** → 排队延迟↓（负载分散），但通信延迟↑

这就是为什么部署需要"权衡"（trade-off），也是 [[gstc-zero-cost-proxy-nas\|GSTC]] 引入"服务-服务器亲和机制"要解决的问题。

## 3. 优化问题的建模与求解

### 3.1 通用建模框架

```
最小化：   f(部署, 路由, 资源分配)
         = α·延迟 + β·成本 + γ·能耗

约束：   - 每个节点的资源不超过容量
         - 每个服务至少有一个实例
         - 端到端延迟不超过 SLA
         - 请求成功率 ≥ 阈值
```

这通常被建模为 [[混合整数非线性规划 (MINLP)]]：
- **整数变量**：是否在节点 v 部署服务 m 的实例（0/1 决策）
- **连续变量**：分配的计算资源量、路由概率

> ⚠️ MINLP 是 NP-hard 的，无法在多项式时间内精确求解。所有论文用的都是**近似/启发式方法**。

### 3.2 求解算法谱系

```
                    求解方法
                       │
        ┌──────────────┼──────────────┐
     精确方法      启发式算法       机器学习方法
     （小规模）                    （新兴）
        │              │               │
     分支定界    ┌──────┼──────┐     强化学习
     暴力搜索    │      │      │     （PPO, DQN）
               贪心   分阶段  元启发式
               算法   算法   （GA, PSO）
                │      │
             资源分割 先部署后路由
             优先放置 （GMDA-RMPR）
```

**三篇论文用到的算法**：

| 算法 | 类型 | 思路 | 论文 |
|------|------|------|------|
| **GMDA** | 贪心 | 服务按实例数降序，部署到平均距离最短的节点 | TPDS 2023 |
| **RMPR** | 概率路由 | 分割请求流，按概率映射到实例 | TPDS 2023 |
| **JCQDA** | 排队感知 | 用 M/M/C 模型指导部署，平衡通信与排队延迟 | GSTC |
| **LAMRA** | 动态替换 | 监控负载，用 Pareto 前沿选模型替换 | GSTC |
| **DA** | 匹配理论 | 延迟接受算法，双边匹配任务与节点 | TSC 2024 |
| **MAB** | 在线学习 | 多臂老虎机探索最优模型放置 | TSC 2024 |
| **Lyapunov** | 在线优化 | 将长期问题分解为短期确定性子问题 | TSC 2024 |

### 3.3 两阶段启发式：最实用的范式

当问题太复杂无法一步求解时，最常用的策略是分两步：

```
阶段1: 部署（Placement）
  输入: 服务需求、节点资源、网络拓扑
  输出: 每个服务部署在哪些节点、多少实例
  方法: 贪心、装箱算法（FFD）、资源分割

阶段2: 路由（Routing）
  输入: 部署方案、请求到达率
  输出: 请求到实例的路由概率
  方法: 概率匹配、最短路径、负载均衡
```

[[joint-deployment-request-routing-microservice-tpds\|TPDS 2023]] 的 GMDA-RMPR 就是这个范式。好处是每步都能求解一个相对简单的子问题，但代价是失去了全局最优性。

### 4.2 从传统微服务到 AI 微服务

传统微服务和 AI 微服务的关键区别：

| 维度 | 传统微服务 | AI 微服务 |
|------|-----------|----------|
| 性能瓶颈 | 业务逻辑处理 | 模型推理计算（GPU 密集） |
| 可扩展性 | 水平扩容（加容器） | 加容器 + 换模型架构 |
| 异构性 | CPU 为主 | CPU + GPU + TPU 混合 |
| 评估指标 | 吞吐量、成功率 | 推理准确率 + 延迟 + 资源 |
| 动态优化 | 扩缩容实例数 | 扩缩容 + 模型替换（[[gstc-zero-cost-proxy-nas\|GSTC]] 的 LAMRA） |

## 5. 做新文章需要了解的关键技术点

### 5.1 你需要掌握的基础

- **排队论基础**：M/M/1、M/M/C、Jackson 网络 — 用它们建模延迟
- **优化建模**：MINLP 的建模方式 — 把部署/路由/资源写成目标函数+约束
- **网络拓扑**：图论基础 — 节点、边、距离、路径

### 5.2 你可以选择的创新方向

| 方向 | 现有不足 | 可能的切入点 |
|------|---------|-------------|
| **更精细的延迟建模** | 大多假设指数分布服务时间 | 考虑异构服务时间、尾部延迟 |
| **更好的联合优化** | 两阶段启发式失去全局最优 | 交替优化、ADMM、强化学习端到端 |
| **动态场景适应** | 大多假设负载已知或平稳 | 在线学习、预测驱动的预部署 |
| **AI 特有挑战** | 模型选择与部署耦合不足 | 联合模型选择+部署+路由 |
| **真实系统验证** | 大多只有仿真实验 | 在 K8s/K3s 上实现原型 |
| **跨层优化** | NAS 和部署分离 | [[gstc-zero-cost-proxy-nas\|GSTC]] 开了个头但尚有空间 |

### 5.3 常用基线算法（写论文时用来对比）

| 算法 | 类型 | 说明 |
|------|------|------|
| **Random** | 随机 | 随机部署+随机路由，最弱基线 |
| **Greedy** | 贪心 | 每步选局部最优 |
| **FFD** | 装箱 | First Fit Decreasing，经典装箱算法 |
| **RLS** | 局部搜索 | Random Local Search，随机邻域搜索 |
| **PSO** | 元启发式 | 粒子群优化 |
| **GA** | 元启发式 | 遗传算法 |
| **PPO/DQN** | 强化学习 | 近年热门基线 |
| **Kubernetes 默认调度** | 工业基线 | 对比工业实际方案 |

### 5.4 常用评估指标

| 指标 | 含义 | 越低/越高越好 |
|------|------|-------------|
| 端到端延迟（E2E Latency） | 请求从进入到响应的总时间 | ↓ |
| 部署成本（Deployment Cost） | 占用的处理器/容器数量 | ↓ |
| 请求成功率（Request Success Rate） | 在 SLA 内完成的请求比例 | ↑ |
| 资源利用率（Resource Utilization） | 已用资源/总资源 | ↑ |
| GPU 显存消耗 | AI 场景特有 | ↓ |
| 能耗（Energy Consumption） | 通信+计算能耗 | ↓ |
| 算法运行时间 | 求解效率 | ↓ |
