# LLM 知识库 — 结构规范（Schema）

基于 Karpathy 的 [LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 模式，由 Claude Code 维护的个人知识库。

---

## 三层架构

| 层级         | 路径           | 所有者          | 说明                            |
| ---------- | ------------ | ------------ | ----------------------------- |
| **原始来源**   | `raw/`       | 人（对 LLM 只读）  | 原始文档：文章、论文、书籍、图片、数据文件。不可变。    |
| **知识库**    | `wiki/`      | LLM          | 生成的 Markdown：摘要、实体页、概念页、综合论述。 |
| **图片资源**   | `asset/img/` | 人 + LLM      | 知识库笔记中嵌入的图片，按笔记名分文件夹存放。       |
| **LLM 对话** | `llm_talk/`  | 人 + LLM      | 与 LLM 对话中产生的灵感、想法、讨论记录。       |
| **结构规范**   | `CLAUDE.md`  | 人 + LLM 共同演进 | 本文件。定义结构、约定和工作流。              |

## 目录结构

```
llm-wiki/
├── CLAUDE.md              # 本规范文件
├── raw/                   # 原始来源（只读）
│   ├── articles/          # 文章、博客
│   ├── papers/            # 论文
│   ├── books/             # 书籍章节
│   └── media/             # 图片、音频、视频
├── wiki/                  # LLM 维护的知识库
│   ├── index.md           # 内容导向的全库目录
│   ├── overview/          # 高层综合、论点、知识状态
│   ├── sources/           # 单个来源的摘要
│   ├── entities/          # 人物、组织、产品、地点
│   ├── concepts/          # 思想、理论、技术、定义
│   └── comparisons/       # 对比页面、表格
├── asset/                 # 知识库引用的资源（图片等）
│   └── img/               # 图片，按笔记名分文件夹
│       └── <笔记名>/      # 例如 attention-mechanism/
│           ├── fig-1.png  # 描述性图片名
│           └── arch.png
├── llm_talk/              # LLM 对话产生的灵感与想法
│   └── YYYY-MM-DD-主题.md # 按日期+主题命名
└── log/                   # 时间线日志（按天存放）
    └── YYYY/
        └── MM-DD.md
```

## 命名约定

- **文件**：使用小写横线连接或描述性标题。Wiki 链接必须与文件名完全匹配。
  - 好：`attention-mechanism.md`、`OpenAI.md`
  - 避免：`file1.md`、`note.md`、`New Document.md`
- **Wiki 链接**：使用 `[[页面名]]`。链接文本尽量简洁。
- **标签**：使用 `#标签` 标识跨类别属性。常用标签：
  - `#待处理` — 未处理的新内容
  - `#草稿` — 需要扩展
  - `#矛盾` — 已标记的冲突
- **日期**：使用 ISO-8601（`YYYY-MM-DD`），放在 frontmatter 和日志条目中。

## 图片资源管理

知识库中的图片（截图、图表、示意图等）统一存放在 `asset/img/` 下，**按笔记名分文件夹**，避免图片散落混乱。

### 目录规则

- **位置**：根目录的 `asset/img/`，与 `wiki/`、`raw/` 同级。
- **文件夹名**：与笔记文件名（不含 `.md`）完全一致。
  - 笔记 `wiki/concepts/attention-mechanism.md` → 图片放 `asset/img/attention-mechanism/`
  - 笔记 `wiki/sources/attention-is-all-you-need.md` → 图片放 `asset/img/attention-is-all-you-need/`
- **图片命名**：使用描述性英文小写名，避免空格。
  - 好：`fig-1-architecture.png`、`loss-curve.png`、`benchmark-table.png`
  - 避免：`image1.png`、`屏幕截图 2026-05-06.png`、`未命名.png`
- **格式**：优先 `.png`、`.jpg`、`.svg`、`.webp`、`.gif`。

### 引用语法

在笔记中使用 Obsidian 原生 wikilink 嵌入：

```markdown
![[asset/img/attention-mechanism/fig-1-architecture.png]]
```

如需描述/标题，使用标准 Markdown 语法（相对路径，便于其他工具识别）：

```markdown
![Transformer 架构示意](../../asset/img/attention-mechanism/fig-1-architecture.png)
*图 1：Transformer 编码器-解码器架构*
```

### 工作流要点

1. **创建笔记时**：若笔记会包含图片，在 `asset/img/<笔记名>/` 下建好同名文件夹（即使初始为空）。
2. **添加图片时**：
   - 从 `raw/media/` 或外部下载的图片，复制（不移动）到对应 `asset/img/<笔记名>/` 下。
   - 重命名为描述性名称。
   - 在笔记中通过 wikilink 引用。
3. **重命名笔记时**：必须同步重命名 `asset/img/<笔记名>/` 文件夹，并修复笔记内的图片引用。
4. **删除笔记时**：检查 `asset/img/<笔记名>/` 是否还被其他笔记引用；若无引用则一并删除。
5. **跨笔记复用图片**：如果同一张图被多个笔记引用，优先放在最相关的那个笔记目录下，其他笔记通过完整路径引用；不要复制副本。

### 清理时检查

在第 3 节"清理"工作流中，额外检查：

- **孤立图片**：`asset/img/` 下未被任何笔记引用的图片或空文件夹。
- **断图链接**：笔记中引用了不存在的图片路径。
- **不规范命名**：含中文、空格、`未命名*`、`image\d+` 等的图片文件。

## 工作流

### 1. 摄入（添加新来源）

当新来源放入 `raw/` 后：

1. **阅读** 来源全文。
2. **讨论** 关键要点与用户确认。
3. **创建** `wiki/sources/` 中的摘要页，遵循来源摘要模板。
4. **处理图片**（如来源含关键图表）：
   - 在 `asset/img/<摘要笔记名>/` 下建文件夹。
   - 从 `raw/` 复制相关图片到该文件夹，重命名为描述性名称。
   - 在摘要页中用 `![[asset/img/<笔记名>/<图>.png]]` 嵌入。
5. **更新** `wiki/index.md` — 将新来源加入目录。
6. **创建或更新** `wiki/entities/` 中的实体页（人物、组织、产品）。
7. **创建或更新** `wiki/concepts/` 中的概念页（思想、技术）。同样按需在 `asset/img/<概念笔记名>/` 下管理图片。
8. **更新** `wiki/overview/` 中的相关综合页（如果来源改变了整体论点）。
9. **记录日志** 到 `log/YYYY/MM-DD.md`，格式：
   ```markdown
   ## [YYYY-MM-DD HH:MM] 摄入 | 来源标题
   - 新增：[[来源标题]]
   - 更新实体：[[实体A]]、[[实体B]]
   - 更新概念：[[概念X]]
   - 新增图片：asset/img/<笔记名>/（N 张）
   - 备注：简短说明
   ```

单个来源可能涉及 10–15 个知识库页面。到处更新交叉引用和 wiki 链接。

### 2. 查询（回答问题）

当用户提问时：

1. **读取** `wiki/index.md` 找到相关页面。
2. **读取** 相关知识库页面（除非明确要求，否则不读原始来源）。
3. **综合** 答案，并引用知识库页面和来源。
4. **可选归档** — 如果答案代表了有价值的新知识（对比、分析、新发现），将其作为新页面归档到知识库。
5. **记录日志**：
   ```markdown
   ## [YYYY-MM-DD HH:MM] 查询 | 问题摘要
   - 归档为：[[新页面]]（或"未归档"）
   - 查阅的关键页面：[[页面A]]、[[页面B]]
   ```

### 3. 清理（健康检查）

定期（或应请求）扫描知识库，检查：

- **页面间矛盾** → 标记 `#矛盾`
- **过时主张** — 已被新来源取代 → 更新或标记
- **孤立页面** — 没有入站链接 → 添加交叉引用或考虑删除
- **缺失概念页** — 重要概念被提及但尚未定义
- **断链** — 指向不存在页面的 wiki 链接
- **信息缺口** — 可通过网络搜索填补
- **孤立图片 / 断图链接** — 见上文"图片资源管理 → 清理时检查"。

记录清理结果：
```markdown
## [YYYY-MM-DD HH:MM] 清理 | 健康检查
- 发现矛盾：N 处
- 孤立页面：N 个
- 过时主张：N 个
- 孤立图片：N 张 / 断图：N 处
- 建议新建页面：[[概念X]]、[[实体Y]]
```

## 日志格式（`log/YYYY/MM-DD.md`）

追加式记录。每日一个文件，存放在 `log/YYYY/MM-DD.md`。

```markdown
# YYYY-MM-DD 日志

## [HH:MM] 摄入 | 来源标题
...

## [HH:MM] 查询 | 问题摘要
...

## [HH:MM] 清理 | 健康检查
...
```

技巧：`ls log/2026/ | sort` 可快速浏览某年的所有日志日。

## 原则

1. **知识库是复利资产。** 每个来源都让它更丰富。每个查询都能增加价值。
2. **积极交叉引用。** Wiki 链接是连接的纽带。
3. **显式标记矛盾。** 不要隐藏冲突 —— 把它们呈现出来。
4. **保持索引最新。** 在中等规模（~100 来源、~数百页面）下，它能替代基于嵌入的 RAG。
5. **为未来的你而写。** 简洁，但包含足够上下文能独立存在。
6. **优先更新现有页面，而非创建新页面。** 避免碎片化。

## 演进

本规范不是静态的。随着知识库增长和领域清晰化：

- 添加新章节或页面类型
- 根据实际效果精炼模板
- 记录新工作流
- 调整约定

更新 `CLAUDE.md` 以反映现实，而非理想。
